350 /*
351  * CLang 4.2 and memory allocation.
352  *
353  * Clang 4.2 has done some optimizations to their memory management that actually causes slightly different behavior than what the C++ Standard defines.
354  * Usually this is not a problem... but in this case, it is a problem.
355  *
356  * More information about the optimization can be found at: http://www.open-std.org/jtc1/sc22/wg21/docs/papers/2012/n3433.html
357  * We've done a bug-report to clang to fix some of this non-standard behavior, which is open at: http://llvm.org/bugs/show_bug.cgi?id=15541
358  *
359  * I very much hope that nobody would actually ever hit this bug/optimization as it is hard to figure out what is going on.
360  *
361  * The original test simply did "new char". Because the memory wasn't assigned to anything and is local in context, the optimization *doesn't* call
362  * the operator new overload. Because it doesn't call the operator new (optimizing away a call to operator new), therefore the method wouldn't throw an exception
363  * and therefore this test failed.
364  *
365  * The first attempt to fix this is to create a local variable and assigned the memory to that. Also this doesn't work as it still detects the allocation is
366  * local and optimizes away the memory call.
367  *
368  * Now, we assign the memory on some static global which fools the optimizer to believe that it isn't local and it stops optimizing the operator new call.
369  *
370  * We (Bas Vodde and Terry Yin) suspect that in a real product, you wouldn't be able to detect the optimization and it's breaking of Standard C++. Therefore,
371  * for now, we keep this hack in the test to fool the optimizer and hope nobody will ever notice this 'optimizer behavior' in a real product.
372  *
373  */
374
