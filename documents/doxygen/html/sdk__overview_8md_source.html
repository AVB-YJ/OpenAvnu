1 EAVB SDK Overview {#sdk_overview}
2 =================
3 
4 
5 Introduction {#sdk_overview_introduction}
6 ============
7 Harman International Industries, Inc. has developed a software Protocol Stack to
8 support Ethernet Audio Video Bridging (EAVB or AVB) on a variety of platforms.
9 The software includes explicit Operating System and Hardware Abstraction Layers
10 (OSAL and HAL) to facilitate rapid and efficient porting of the generic stack to
11 specific platforms. The complete OPENAVB EAVB implementation, and its role within an
12 overall EAVB system are discussed in OPENAVB?s Ethernet Audio Video Bridging (AVB)
13 High Level Specification (OPENAVB13-01059).
14 
15 The OPENAVB AVB stack is designed to work on both General Purpose OSes (GPOS), such
16 as Linux, as well as Real-Time OSes (RTOS). There are some difference in
17 concepts and terms between a GPOS and RTOS, for example threads vs tasks, or the
18 concept of multiple processes. For the purposes of the SDK guides the terms
19 thread and task are used interchangeably.
20 
21 The guides in this SDK focus on integrating the OPENAVB AVB stack in an application,
22 developing interface module components and configuration AVB streams. These
23 guides are separated into four sections:
24 
25 - [EAVB SDK Overview](@ref sdk_overview)
26 - [EAVB Integration](@ref sdk_integration)
27 - [AVTP Interface Module Development](@ref sdk_avtp_interface_module_dev)
28 - [AVTP Stream Configuration](@ref sdk_avtp_stream_cfg)
29 
30 This overview section will cover general architecture.

34 Glossary {#sdk_overview_glossary}
35 ========
36 **AAF:** AVTP Audio Format
37 
38 **AVB:** Audio Video Bridging (used interchangeably with EAVB)
39 
40 **AVB Stack:** The primary functionality that implements the various AVB
41 protocols
42 
43 **AVTP:** Audio Video Transport Protocol
44 
45 **EAVB:** Ethernet Audio Video Bridging (used interchangeably with AVB)
46 
47 **EMAC:** Ethernet Media Access Control
48 
49 **FQTSS:** Forwarding and Queuing enhancements for Time Sensitive Streams
50 
51 **GPOS:** General purpose OS
52 
53 **gPTP:** generalized Precision Time Protocol
54 
55 **HAL:** Hardware Abstraction Layer
56 
57 **Listener:** Receives an AVTP stream, unpacks it and pushes it to a media sink
58 for playback. It consists of a dedicated task and uses functionality in the AVTP
59 module, one mapping module and one interface module.
60 
61 **IEEE:** Institute of Electrical and Electronics Engineers
62 
63 **Interface Module:** An AVTP component that when called from the talker pulls
64 data from a media source and pushes it onto the media queue in the format
65 expected by the mapping module running on the talker. When called from a
66 listener it pulls data from the media queue and pushes it to the media sink for
67 playback.
68 
69 **ISR:** Interrupt Service Routine
70 
71 **MAAP:** Multicast Address Allocation Protocol
72 
73 **MAC:** Media Access Control
74 
75 **Mapping Module:** An AVTP component that when called from the talker pulls
76 data from the media queue and packages it into the AVTP data payload according
77 to a specific AVB encapsulation. When called from a listener it unpacks the AVTP
78 data payload according to the specific AVB encapsulation and pushes it on to the
79 media queue.
80 
81 **Media Queue:** A circular FIFO container used to opaquely pass media data
82 blocks between interface modules and mapping modules. This is the only way
83 interface modules and mapping modules communicate.
84 
85 **MJPEG:** Motion Joint Photographic Expert Group
86 
87 **MPEG2-TS:** Motion Picture Expert Group (version 2) Transport Stream
88 
89 **OS:** Operating System
90 
91 **OSAL:** Operating System Abstraction Layer
92 
93 **SDK:** Software Development Kit
94 
95 **SR:** Stream Reservation
96 
97 **RTOS:** Real-time OS
98 
99 **SRP:** Stream Reservation Protocol
100 
101 **Stream:** A series of AVTP data packets
102 
103 **Talker:** Takes an audio or video source and transmits it as an AVTP stream on
104 the AVB network. It consists of a dedicated task and uses functionality in the
105 AVTP module, one mapping module and one interface module.

181 Integration, Extension and Configuration {#sdk_overview_integration}
182 ========================================
183 Three distinct uses of the SDK are as follows.
184 
185 **Integration:** This refers to the integration of the OPENAVB AVB stack into an
186 existing or new application. This is also sometimes referred to as the hosting
187 application. For details related to this see [EAVB Integration](@ref
188 sdk_integration)
189 
190 **AVTP Interface Module Development:** In order for the AVB stack to be used it
191 must have interaction with media data sources and sinks on the platform. This is
192 accomplished with Interface modules. The AVB stack ported to a platform may
193 include some interface modules already but typically new interface modules are
194 needed for the specific hardware and drivers being targeted. See [AVTP Interface
195 Module Development](@ref sdk_avtp_interface_module_dev) for more details.
196 
197 **Configuration:** Configuration can be split into 2 distinct area. Hosting
198 application configuration and stream configuration. Hosting application
199 configuration is port specifici and therefore not covered in detail in this SDK.
200 However, in the configuration guide section of the SDK some host application
201 configuration example are provided as a point of reference. The other primary
202 area of configuration is related to stream. See the section [AVTP Stream
203 Configuration](@ref sdk_avtp_stream_cfg) for more details.
204

207 Platform Specific Considerations {#sdk_overview_platform}
208 ================================
209 The OPENAVB AVB stack is designed for portability covering both general purpose OSes
210 and real-time OSes. The base public APIs of the SDK do not change depending on
211 the port. However, some elements of working with the SDK do change depending on
212 the port of the OPENAVB stack. See the release notes for the specific port for
213 details that may be beyond the common SDK usage.
214 
215 ## Threads / Tasks
216 Within the SDK guides the terms threads and tasks are used interchangeability
217 and for the purposes of the SDK they can be considered the same.
218 
219 ## Processes
220 Some platform OSes support multiple processes. The OPENAVB stack for a particular
221 port may make use of multiple processes. For example in the Linux reference
222 implementation gPTP resided in a separate process. Additionally in this
223 reference implementation the talk / listener end-station functionality for each
224 stream can be in a single process or split across multiple processes. Whereas in
225 a RTOS there isn't typically the concept of multile processes.
226 
227 ## Configuration Overview
228 Configuration of both the host application as well as the streams can vary based
229 on platform capabilities. For example on configuration information may come from
230 .ini files on the device and in other platforms there may not be a file system
231 in which case configuration may be set at build time.
232 
233 ## Startup
234 See the port specific Release Notes for the details on AVB start up.
235 
236 ## Libraries
237 The AVB core stack commonly is build as a static library and linked with a
238 hosting application. Interface modules may be built differently depending on the
239 platform. In some cases they my be dynamic libraries and other cases they may be
240 static libraries. It is also possible that the interface modules will be built
241 with the host application or as part of the AVB core library.
