1 AVTP Interface Module Development {#sdk_avtp_interface_module_dev}
2 =================================
3 
4 Introduction {#sdk_avtp_intf_module_introduction}
5 ============
6 Interface modules are the components that sit between the core AVB stack and the
7 platform device drivers that supply access to media hardware.
8 
9 <br>
10 
11 The Plug-in Architecture {#sdk_avtp_intf_module_plugin}
12 ========================
13 The OPENAVB AVB stack has a plug-in architecture for the AVTP implementation in the
14 form of interface modules. This allows for easily hooking into the AVB stack to
15 extend it for interfacing to media specific hardware.
16 
17 Mapping modules, which are mentioned through-out these guides, are also
18 plug-ins. These implement the various AVTP encapsulations and require a deep
19 understanding of AVTP and therefore are developed internally by OPENAVB.
20 
21 Interface modules are discovered at runtime using configuration information
22 that is passed to the openavbTLOpen() function. Below is an example of a section of
23 the configuration information with interface module values.
24 
25 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
26 
27 // pIntfInitFn : Interface module initialization function.
28 // openavb_intf_initialize_fn_t)
29 cfg->osalCfg.pIntfInitFn = openavbIntfJ6Video;
30 
31 ///////////////////
32 // Interface module
33 ///////////////////
34 
35 // intf_nv_blocking_tx_callback : A talker only option. When set packet pacing
36 // is expected to be handled in the interface module.
37 // Commonly the TL configuration option tx_blocking_in_intf needs to be set
38 // to match this.
39 // cfg->libCfgNames[cfgIdx] = "intf_nv_blocking_tx_callback";
40 // cfg->libCfgValues[cfgIdx++] = "1";
41 
42 // intf_nv_ignore_timestamp : If set the listener will ignore the timestamp
43 // on media queue items.
44 cfg->libCfgNames[cfgIdx] = "intf_nv_ignore_timestamp";
45 cfg->libCfgValues[cfgIdx++] = "0";
46 
47 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
48 
49 The function pointer variable `pIntfInitFn` points to the interface module
50 initialization function.
51 
52 When the talker/listener module calls the interface module initialization
53 function it passes in a callback structure as a parameter that must get filled
54 with the required callback function addresses by the interface module.
55 
56 After initialization the talker/listener module will call directly into the
57 interface module via these callbacks. See the API reference section for details
58 and examples of these callbacks.
59 
60 The callback functions in an interface module, aside from the initialization
61 call, receives a pointer to a media queue structure. This structure is
62 self-contained, similar to an object in C++, in that it includes a function
63 callback list for the interface module to call into the media queue to access
64 and work with the media queue.
65 
66 <br>
67 
68 Task / Thread Model {#sdk_avtp_intf_module_task}
69 =================
70 In the general case the task / threading (hereafter referred to as task) model
71 for interface modules is very simple. All callbacks will occur on the same task
72 and therefore there are no synchronization concerns. However, access to media
73 sources and media sinks can have demanding timing and processing requirements
74 and it may be necessary for an interface module to have complete control over an
75 execution task to ensure timely pulling or pushing data into the media hardware
76 or driver. In those cases the interface module may find it a benefit to create
77 its own task or tasks.
78 
79 When an interface module has its own task the point of synchronization between
80 the callback thread and the interface module private thread would be the
81 sharing of a media queue item pointer. When a head or tail item is locked a
82 pointer to that media queue item is returned. At that point the data area for
83 that item remains locked and can be used by another task beyond the scope of
84 the callback into the interface module. The media queue item will remain locked
85 until unlocked (or pushed or pulled). The media queue itself can't be locked,
86 it can only be accessed within the scope of a callback into the interface
87 module.
88 
89 <br>
90 
91 Building an Interface Module {#sdk_avtp_intf_module_building}
92 ============================
93 There should be minimal dependencies for creation of an interface module. The
94 interface module can be built as a static library and include all the required
95 callbacks as defined in the reference section of this document.
96 
97 Access to all functionality in the AVB stack from an interface module is also
98 handled via callbacks that are available to the interface module in the media
99 queue structure it receives as a parameter on its own callbacks.
100 
101 <br>
102 
103 Interface Module in a Talker {#sdk_avtp_intf_module_talker}
104 ============================
105 An interface module when used in a talker pulls data from a media source and
106 pushes it onto the media queue in the format expected by the mapping module
107 configured for that stream.
108 
109 <br>
110 
111 Interface Module in a Listener {#sdk_avtp_intf_module_listener}
112 ==============================
113 When called from a listener it pulls data from the media queue and pushes it to
114 the media sink for presentation.
115 
116 <br>
117 
118 Working With the Media Queue {#working_with_mediaq}
119 ============================
120 The media queue is the conduit between interface modules and mapping modules.
121 The media queue is the only entry point for interface modules to call into the
122 AVB stack and allows for pushing or pulling media data though it as well as
123 access to time functionality in the AVTP time structure.
124 
125 The media queue is internally implemented as a circular FIFO container. The
126 format of the data for items in the media queue is defined by the mapping
127 module being used in the talker/listener. For some mapping modules the data
128 format of an item will match the defined AVB encapsulation. For example the
129 MPEG2-TS (61883-4) mapping module data format of media queue items are simple
130 192 byte MPEG2-TS source packets. Each media queue item contains one or more
131 MPEG2-TS source packet. The actual MPEG2-TS mapping may combine multiple source
132 packets into one AVTP packet but that is hidden from the interface module.
133 
134 The media queue functions are accessed as API calls.
135 
136 For example, the openavbMediaQHeadLock() function is called as:
137 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
138  openavbMediaQHeadLock(pMediaQ);
139 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
140 The pMediaQ pointer is passed to all of the interface module callbacks.
141 
142 Access to the queue items is accomplished with the Head functions for pushing
143 data into the media queue. These are used by interface modules running in a
144 talker. To read items from the media queue the Tail functions are used by
145 interface modules running in a listener.
146 
147 At any one time only 2 items are accessible in the media queue. These are the
148 head item and tail item. Care must be taken to always match up the
149 openavbMediaQHeadLock() with a openavbMediaQHeadUnlock() or openavbMediaQHeadPush()
150 functions. The same applies that the openavbMediaQTailLock() must be paired with
151 either a openavbMediaQTailPull() or openavbMediaQTailUnlock() functions. This means
152 a new item can not be added to the media queue until a openavbMediaQHeadPush()
153 is called and a new item can not be pulled from the media queue until
154 openavbMediaQTailPull() is called.
155 
156 For a detailed work flow please visit
157 [Media Queue Usage](@ref sdk_notes_media_queue_usage)
158 
159 <br>

161 Timestamps {#sdk_avtp_intf_module_timestamps}
162 ==========
163 For interface modules often the only requirement for timestamps is to assign a
164 PTP time to the items added to the media queue when running as a talker. For
165 example:
166 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
167  openavbAvtpTimeSetToWallTime(pMediaQItem->pAvtpTime);
168 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
169 In most cases the mapping modules and media queue take care of the rest of the
170 requirements for AVTP.
