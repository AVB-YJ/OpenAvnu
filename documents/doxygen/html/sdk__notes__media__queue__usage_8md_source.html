1 Media Queue Usage {#sdk_notes_media_queue_usage}
2 =================
3 
4 # Description
5 
6 This section presents the work flow required for working with the Media Queue.
7 Please note that most of the described steps are performed inside the AVB stack
8 and are hidden from interface module implemented. There are also some
9 simplifications to make description as straightforward as possible.
10 
11 <br>
12 
13 # Workflow
14 
15 The following work flow steps will be described.
16 * [Starting](@ref media_queue_usage_start) - Initialization common for talker
17 and listener streams
18 * [Talker specific](@ref media_queue_usage_talker) - Talker specific usage
19 * [Listener specific](@ref media_queue_usage_listener) - Listener specific usage
20 * [Stopping](@ref media_queue_usage_stop) - Stopping procedure common for talker
21 and listener streams
22 * [Rules](@ref media_queue_usage_rules) - general rules to follow while using
23 the Media Queue
24 
25 <br>
26 
27 Starting {#media_queue_usage_start}
28 ========
29 
30 First the Media Queue has to be created and initialized with correct data. Below
31 are the function calls needed for proper creation and initialization. Note:
32 these calls are initiated from the AVTP module.
33 
34 * @ref openavbMediaQCreate - data is allocated, queue is initialized, internal data
35 structures are prepared
36 * @ref openavbMediaQSetMaxStaleTail - sets maximum stale in microseconds before data
37 is being purged
38 * The Interface module initialization function for this media queue is called
39 with media queue as a parameter, those functions
40  * Set internal interface module parameters
41  * Initialize Media Queue
42  * Creates Media Queue Interface Module Private Data
43  * Fills the private data structure with information needed by interface
44  module
45 * Stream configuration values are processed. This actually configuration data
46 may come from .ini files or internally set within the AVB host application.
47 * Mapping module init function openavb_map_cb_t::map_gen_init_cb is being called
48 during which several steps (media queue parameters **have to** be set)
49  * Media Queue Items count is set
50  * Media Queue Items are allocated
51 * Interface module init function openavb_intf_cb_t::intf_gen_init_cb function is
52 called
53 
54 Now the listener/talker stream is running. See next steps below for details of
55 Media Queue interaction for the talker and listener.
56 
57 <br>
58 
59 Talker specific flow {#media_queue_usage_talker}
60 ====================
61 
62 As a talker, an interface module is responsible for writing data to Media Queue,
63 so important steps are:
64 
65 * @ref openavbMediaQHeadLock function for getting the head of the media queue and
66 marking it as locked
67 * @ref openavbMediaQHeadUnlock function which releases head. Any subsequent calls to
68 @ref openavbMediaQHeadLock will return the same MediaQueueItem
69 * @ref openavbMediaQHeadPush function unlocks head previously taken by the Lock
70 function and informs that work on the current Item is finished so that next call
71 to @ref openavbMediaQHeadLock will return next Media Queue Item. Call of this
72 function makes the item accessible via the @ref openavbMediaQTailLock function. This
73 function additionally unlocks the head so the @ref openavbMediaQHeadUnlock call is
74 not needed
75 
76 <br>
77 
78 Listener specific flow {#media_queue_usage_listener}
79 ======================
80 
81 As a listener, an interface module works on Media Queue tail elements. Data in
82 those items is being written by the mapping module. These are the key Media
83 Queue functions for the listener functionality in an interface module:
84 
85 * @ref openavbMediaQTailLock gets an item from the tail and allows working on
86 the current tail item of the Media Queue
87 * @ref openavbMediaQTailUnlock unlocks the tail element in MediaQueue which means
88 that interface module has stopped working on it for now but processing of
89 current tail element will be continued later
90 * @ref openavbMediaQTailPull unlocks the tail element and removes it from the Media
91 Queue. This means that the interface module has finished processing of current
92 tail element and it can be rewritten again by new data. This function
93 additionally unlocks tail, so it is not necessary to call @ref
94 openavbMediaQTailUnlock.
95 
96 <br>
97 
98 Stopping {#media_queue_usage_stop}
99 ========
100 
101 During the stopping process following action are taken
102 
103 * openavb_intf_cb_t::intf_gen_end_cb is called
104 * openavb_map_cb_t::map_gen_end_cb is called
105 * Media Queue is deleted using the @ref openavbMediaQDelete function which frees
106 the memory taken by all internal structures of Media Queue
107 
108 <br>
109 
110 Guidelines {#media_queue_usage_rules}
111 ==========
112 
113 * Calls to **Lock** functions must always be paired with their **Unlock**
114 counterparts to avoid problems while working with MediaQueue. Push and Pull
115 functions additional result in an unlock.
116 * The current implementation Media Queue allows accessing two elements at the
117 same time - one is head and second the tail
118 * The number of Media Queue items and their sizes depends on configuration for
119 the stream. The main driving factor is to make the data accessible for the next
120 element that follows interface module. The size of the Media Queue item is a
121 factor of the According to this Media Queue number of size of AVTP stream
122 encapsulation payload multiplied by the packing factor. The number of Media
123 Queue items required if different for a talker and listener. The talker usually
124 needs minimal Media Queue items. However, the listener must have enough Media
125 Queue items to buffer data until the AVTP presentation time. This is based on
126 the maximum transit time for the SR Class in use. ring decoding
127 * If there are not enough Media Queue items warnings will be logged to the
128 implemented logging system for the port for debugging purposes.
129
